# Publish Income — Royalty Reconciliation

Compares incoming royalty statements (DSP, distributor, PRO) against an internal catalog ledger, identifies discrepancies, and generates a structured income report ready for accounting review.

---

## Step 1 — Identify the Statement Type

Ask the user (or infer from context):

1. **Source type** — DSP direct (Spotify, Apple, YouTube, Amazon), distributor (DistroKid, TuneCore, CD Baby, Amuse, AWAL), or PRO (ASCAP, BMI, SESAC, PRS, SOCAN)?
2. **Statement period** — month, quarter, or annual?
3. **Format available** — CSV, XLSX, PDF, or raw paste?
4. **Do they have an existing internal ledger?** If yes, request it. If no, proceed to build one from the statement.

Load `references/statement-formats.md` to confirm expected column schema for the identified source.

---

## Step 2 — Parse the Statement

Extract the following fields from the statement. Map non-standard column names to the canonical ledger fields:

| Canonical field | Common aliases |
|---|---|
| `track_title` | Song, Title, Track Name, Recording Title |
| `isrc` | ISRC Code, Recording ID |
| `period_start` | Report Start, Period From, Statement From |
| `period_end` | Report End, Period To, Statement Through |
| `territory` | Country, Region, Market |
| `streams` | Plays, Streams Count, Units |
| `rate` | Per-Stream Rate, Unit Rate, Royalty Rate |
| `gross_revenue` | Gross Royalty, Revenue, Gross Amount |
| `net_revenue` | Net Royalty, Net Payment, Net Amount |
| `source` | Platform, Store, Service |

Flag any rows with missing `isrc`, `track_title`, or `net_revenue` — these cannot be reconciled automatically.

---

## Step 3 — Build or Update the Master Ledger

Load `references/ledger-template.md` for the standard ledger structure.

If no existing ledger exists: populate a new ledger from the statement data.

If an existing ledger is provided:
1. Match statement rows to ledger entries by `isrc` (primary) or `track_title` (fallback).
2. For each matched row, compare `net_revenue` against the ledger's `expected_revenue`.
3. For each unmatched statement row, flag as **NEW** — not in ledger, may indicate an unlicensed use or missing catalog entry.
4. For each ledger entry with no matching statement row, flag as **MISSING** — expected income not received.

---

## Step 4 — Run Discrepancy Analysis

Load `references/reconciliation-logic.md` for threshold rules and discrepancy categories.

Classify each discrepancy:

| Code | Category | Threshold |
|---|---|---|
| `D1` | Minor variance | < 5% of expected |
| `D2` | Moderate variance | 5–20% of expected |
| `D3` | Significant variance | > 20% of expected |
| `D4` | Missing payment | Expected > $0, received $0 |
| `D5` | Unexpected income | No ledger entry, payment received |
| `D6` | ISRC mismatch | Payment received under wrong ISRC |
| `D7` | Territory anomaly | Payment from unexpected territory |

---

---

## Step 6 — Offer Follow-On Actions

After delivering the report:

- Offer to **export the updated master ledger** as a CSV-formatted table.
- If D4 flags exist, offer to **draft a dispute notice** to the distributor or platform.
- Offer to **save the report** as a `.md` file for the accounting log.
- If the user has multiple statements from the same period, offer to **run a multi-source consolidation** (aggregate all DSP + distributor + PRO into a single period total).

---

## Key Principles

- **Never fabricate revenue figures.** If data is ambiguous or missing, flag it rather than estimate.
- **ISRC is the canonical identifier.** Track title matching is a fallback only — flag title-matched rows as lower confidence.
- **Always include the advisory notice** in the output — this is not accounting software.
- **Disputes need human sign-off.** Draft dispute language, but require the user to review before sending.
