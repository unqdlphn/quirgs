# ISO 42001 — Evidence Guide: What Auditors Look For

This guide maps each major audit area to the specific documents, records, and behavioral
evidence that certification auditors typically request. Use this to prioritize what to
prepare before any audit.

---

## The Golden Rule of ISO Audits

> "If it isn't documented, it didn't happen."

Auditors work from objective evidence. Verbal explanations are supplementary — the
primary evidence is always a document, record, or observable process.

---

## Document Types: Definitions

| Type | Definition | Examples |
|---|---|---|
| **Documented information** (documents) | Describes how things should be done | Policies, procedures, guidelines, plans |
| **Documented information** (records) | Proves that things were done | Meeting minutes, training logs, audit reports, completed checklists |

Both types are required. Policies without records = Stage 1 pass, Stage 2 fail.

## Evidence Checklist by Audit Area

### Foundational Documents (Must have before Stage 1)

| Document | Clause | Format |
|---|---|---|
| AIMS Scope Statement | 4.3 | 1-page approved document |
| Context Analysis | 4.1 | Document or section of AIMS manual |
| Stakeholder Register | 4.2 | Table: stakeholder / needs / obligations |
| AI Policy | 5.2 | Signed, dated, version-controlled |
| AI Objectives | 6.2 | Table: objective / metric / target / owner |
| Risk Register | 6.1 | Table: risk / likelihood / impact / treatment / owner |
| Statement of Applicability | Annex A | Table: control / included? / justification / evidence reference |
| AIMS Manual or Framework Document | 4.4 | Overview of AIMS processes and their interactions |

---

### Operational Records (Must have before Stage 2)

| Record | Clause | Format |
|---|---|---|
| AI System Impact Assessments (one per in-scope system) | 8.2, A.5 | Completed template per system |
| Training / competency records | 7.2 | Attendance log, completion certificates |
| Internal audit report(s) | 9.2 | Formal audit report with findings |
| Management review minutes | 9.3 | Signed meeting minutes with agenda items addressed |
| Nonconformity register | 10.2 | Table: NC / root cause / corrective action / status |
| Supplier assessment records | 8.4, A.10 | Assessment form per key AI supplier |
| Performance reports / KPI data | 9.1 | Dashboard or report covering defined metrics |
| Monitoring plan and records | 8.1, A.6.5 | Plan + evidence of monitoring activity |

## Common Audit Findings (and How to Avoid Them)

| Finding | Root cause | Prevention |
|---|---|---|
| AI policy not communicated to staff | Policy published but no awareness program | Run a documented awareness session; keep attendance records |
| No AI system impact assessment | Process defined but not implemented | Complete assessments before Stage 2; use the Annex B categories |
| Objectives not measurable | Vague objectives like "improve AI safety" | Attach a metric and target to every objective |
| Internal audit not conducted | Audit scheduled but not executed | Complete at least one internal audit; document findings |
| Management review not held | Meeting held but not documented | Keep signed minutes; confirm all required inputs were covered |
| Nonconformities not closed | RCA done but actions not tracked | Maintain a live NC register with due dates and verification |
| Supplier risk not assessed | Third-party AI tools not inventoried | Create a supplier register; complete a basic risk assessment per vendor |
| SoA not prepared | Annex A controls not reviewed | Complete the SoA before Stage 1 |

## Auditor Interview Questions (Prepare Answers For These)

### For top management:
- "How do you demonstrate your commitment to responsible AI?"
- "What is your AI risk appetite?"
- "When did you last review AI performance data?"

### For the AIMS manager / AI governance lead:
- "Walk me through your AI system impact assessment process."
- "Show me a completed assessment for your most significant AI system."
- "How do you handle a nonconformity?"
- "What changed in your AIMS since your last review?"

### For AI developers / operators:
- "How do you know what data quality requirements apply to your system?"
- "What would you do if you discovered a bias problem in production?"
- "How do you report an AI incident?"

### For end users of AI systems:
- "Do you know you're using an AI system?"
- "Where do you go if you have a concern about the AI output?"
- "Have you been trained on how to use this system responsibly?"

## Document Control Checklist

Every controlled document should have:
- [ ] Document title and unique ID
- [ ] Version number
- [ ] Date of issue / last review
- [ ] Author / owner
- [ ] Approver (with signature or digital approval record)
- [ ] Review frequency
- [ ] Distribution / access location

Auditors will check these fields. Documents missing version numbers or approval signatures
are a common minor nonconformity.

## Audit Readiness Score (Quick Self-Assessment)

Rate each area 0–2: (0 = not started, 1 = in progress, 2 = complete with evidence)

| Area | Score |
|---|---|
| Scope, context, stakeholder analysis documented | /2 |
| AI policy approved and communicated | /2 |
| Objectives defined with metrics | /2 |
| Risk register current | /2 |
| Statement of Applicability complete | /2 |
| Impact assessments completed | /2 |
| Internal audit conducted | /2 |
| Management review conducted | /2 |
| Nonconformity register maintained | /2 |
| Training records up to date | /2 |

**Score interpretation:**
- 18–20: Stage 2 ready
- 14–17: Stage 1 ready; Stage 2 needs work
- 10–13: Stage 1 borderline; significant gaps
- Below 10: Not ready for external audit; focus on foundational documents first
